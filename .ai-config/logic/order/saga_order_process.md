# Saga 패턴(Choreography) 기반 분산 주문 프로세스

본 문서는 서비스가 분리된 마이크로서비스 아키텍처(MSA)에서 결과적 정합성(Eventual Consistency)을 달성하기 위한 Saga 패턴 설계를 다룹니다.

## 1. 전체 프로세스 흐름 (Choreography Saga)

1.  **주문 서비스: 주문 생성 (Pending)**
    - 주문을 `PENDING` 상태로 저장하고 `OrderCreated` 이벤트를 발행합니다.
2.  **재고 서비스: 재고 차감**
    - 이벤트를 구독하여 재고를 차감한 후, 결과(`StockReserved` 또는 `StockReserveFailed`)를 발행합니다.
3.  **주문 서비스: 상태 업데이트**
    - 성공 시: 주문 상태를 `WAITING_FOR_PAYMENT`로 변경하고 5분 지연 메시지를 발행합니다.
    - 실패 시: 주문을 즉시 취소 처리(보상 트랜잭션)합니다.
4.  **결제 서비스: 결제 처리**
    - 사용자가 결제 완료 시 `PaymentCompleted` 이벤트를 발행합니다.
5.  **타임아웃 처리 (Timeout)**
    - 5분 뒤 지연 메시지가 도착했을 때 여전히 결제 전이면 `OrderCancelled` 이벤트를 발행합니다.
6.  **재고 서비스: 재고 복구 (보상 트랜잭션)**
    - `OrderCancelled` 이벤트를 수신하면 차감했던 재고를 다시 원복합니다.

## 2. 왜 '선 주문 생성'인가?

분산 시스템에서는 전체 과정을 추적할 수 있는 **식별자(Order ID)**가 반드시 필요합니다.
- **책임 소재:** 주문 서비스가 전체 시나리오의 시작점으로서 기록을 남겨야 이후 단계(재고, 결제)에서 실패가 발생해도 추적이 가능합니다.
- **상태 관리:** 각 서비스는 자신의 DB만 책임지므로, 전체 상태는 이벤트를 통해 점진적으로 완성됩니다.

## 3. 보상 트랜잭션 (Compensating Transaction)

Saga 패턴은 롤백이 불가능한 분산 환경이므로, 에러 발생 시 이전에 성공한 작업을 되돌리는 '보상 작업'을 수행합니다.
- 예: 재고 차감 후 결제 실패 시 -> 재고 복구 이벤트를 발행하여 수동으로 수치를 맞춤.

## 4.  (Pro & Con)

- **장점:** 서비스 간 결합도가 낮아지고 확장성이 매우 뛰어납니다. 특정 서비스(예: 재고)가 일시적으로 중단되어도 전체 시스템이 마비되지 않습니다.
- **주의사항:** 로직이 비동기로 흐르기 때문에 디버깅이 어렵습니다. 메시지 중복 수신에 대비한 **멱등성(Idempotency)** 보장 로직이 필수적입니다.
