#  단일 DB 트랜잭션 기반 주문/재고 처리 프로세스

본 문서는 단일 데이터베이스 환경에서 ACID 트랜잭션을 활용하여 데이터 정합성을 최우선으로 하는 주문 처리 설계를 다룹니다.

## 1. 전체 프로세스 흐름 (Flow)

1.  **재고 검증 및 차감 (Atomic Update)**
    - DB의 단일 쿼리(`UPDATE ... WHERE stock > 0`)를 사용하여 원자적으로 재고를 줄입니다.
2.  **주문(Order) 생성**
    - 주문 기본 정보를 `PENDING` 상태로 생성합니다.
3.  **주문 목록(Order Item) 생성**
    - 구매 시점의 상품명, 가격 등을 스냅샷 형태로 저장합니다.
4.  **결제(Payment) 생성**
    - 결제 요청 정보를 `PENDING` 상태로 생성합니다.
5.  **Waiting Queue 메시지 발행**
    - RabbitMQ의 지연 큐(TTL)에 주문 정보를 담아 보냅니다.
6.  **Process Queue 메시지 도착**
    - 설정된 시간(예: 5분)이 지나면 메시지가 실제 처리 큐로 이동합니다.
7.  **타임아웃 컨슈머 처리**
    - 메시지를 수신한 컨슈머가 DB를 조회하여 결제 여부를 확인하고, 미결제 시 주문 취소 및 재고 복구를 수행합니다.

## 2. 트랜잭션 설계 (Transaction Scope)

### 원자적 범위 (1~4단계)
- **ACID 보장:** 1단계(재고 차감)부터 4단계(결제 생성)까지는 하나의 `@Transactional` 블록으로 묶입니다. 
- **롤백 전략:** 만약 재고가 부족하거나 주문 생성 중 에러가 발생하면 전체 로직이 롤백되어 데이터 미스매치를 방지합니다.

## 3. RabbitMQ를 이용한 지연 처리

- **패턴:** TTL(Time-To-Live) + DLX(Dead Letter Exchange)
- **목적:** 결제 대기 시간(5분) 동안 시스템 자원을 점유하지 않고 비동기적으로 타임아웃을 관리합니다.
- **컨슈머 로직:** - `SELECT` 주문 상태 확인 -> `If (PENDING)` -> `UPDATE (CANCEL)` & `Inventory + 1`.

## 4. 시니어의 조언 (Pro & Con)

- **장점:** 구현이 매우 직관적이며 DB 레벨에서 데이터 정합성이 완벽하게 보장됩니다. 초기 서비스나 중소규모 트래픽에서 가장 권장되는 패턴입니다.
- **주의사항:** 재고 테이블에 대한 락(Lock) 시간이 길어질 수 있으므로, 대규모 트래픽 발생 시 성능 병목 지점을 모니터링해야 합니다.
